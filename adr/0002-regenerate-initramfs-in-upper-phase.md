# ADR 0002: Upperフェーズでのinitramfsとカーネルの再生成（instantiate）

## ステータス

**Accepted**

## コンテキスト

genpackのシステムイメージビルドプロセスは、コンパイルと全パッケージのインストールを行う「Lower層」と、そこから実行時に必要なファイルのみを抽出（copy-up）してカスタマイズを施す「Upper層」の2段階（レイヤードアーキテクチャ）で構成されています。

イメージのブートに必須となる `initramfs` やカーネルイメージ本体は、当然最終成果物（SquashFS）に含まれる必要があります。しかし、これらはLowerフェーズでビルドが完了した時点ですでに一度生成されているにもかかわらず、genpackはUpper層のビルドフェーズ初期（`genpack-exec-package-scripts`）において、わざわざ `emerge --config <kernelpkg>` 等を実行し、Upper層内で再度カーネルのデプロイと `initramfs` (dracut) の生成を行っています。

この「毎回再生成する」プロセスは、CPUやディスクI/Oを消費しビルド時間（特にUpperフェーズの反復実行時）のボトルネックとなっています。「単にLowerの `/boot` 以下のエントリをUpperへ自動的にコピーすれば高速化できるのではないか」という疑問が生じました。

## 決定

**「Lower層で生成された `/boot` 以下のファイルを自動コピーして使い回す」という安直な高速化手法は採用せず、現在の「Upperフェーズにおいて、パッケージ固有のスクリプト経由でカーネルのインストール機構（`installkernel` 等）を再度キックし、Upper環境内で正規に実体化（instantiate）させる」アーキテクチャを維持します。**

## 理由（Rationale）

この設計は、Gentooの強力なフックエコシステムとアーキテクチャごとの差異を、genpackのコアロジックから完全に抽象化・隠蔽するための必然的な選択です。

1. **Portage (VDB) 管理外のファイルであるため**
   Gentooの思想として、カーネル本体の `/boot` へのデプロイや `initramfs` の生成は、ebuildのインストール（`src_install`）ではなく、パッケージ管理外の `installkernel` 機構（フックスクリプト）に委ねられています。そのため、ファイルの実体がPortageのパッケージデータベース (VDB) に記録されず、genpackの標準抽出ツールである `genpack-copyup` の対象から必然的に漏れます。

2. **カーネルインストール機構がもたらす「未知の副作用」を許容するため**
   `installkernel` 機構が生成するものは `/boot/vmlinuz` や `/boot/initramfs` だけとは限りません。USEフラグや環境次第で、`/efi` への **EFIスタブ** や **UKI (Unified Kernel Image)** の配置、あるいは `systemd-boot` 用の設定ファイル（`/boot/loader/entries/*.conf`）などを動的に生成する可能性があります。
   「`/boot` ディレクトリだけをコピーすれば良い」とgenpackコアにハードコードしてしまうと、これらの正規のフックがもたらす恩恵をすべて破壊してしまいます。

3. **プラットフォーム固有のデプロイ挙動の隠蔽**
   例えば、Raspberry Pi 用のカーネル (`raspberrypi-image`) は `virtual/dist-kernel` に属さず、独自のパッケージスクリプトで `dracut` を叩き、さらに `/boot/config.txt` に `auto_initramfs=1` を動的に追記するというハードウェア固有の副作用を持っています。
   Upperフェーズで「各パッケージが定義した正規の手順」を再実行させることで、genpack本体は「PC向けか、Raspberry Pi向けか」を一切意識することなく、正しいブート環境をUpper層に構築させることができます。

## 結果（Consequences）

*   **Pros (利点):** 新しいブート方式（UKIなど）や未知のアーキテクチャへの対応力が極めて高く保たれます。genpackのコアビルダーにプラットフォーム固有のハック（パッチ）を埋め込む必要がありません。
*   **Cons (欠点):** Upperフェーズをビルドするたびに `dracut` が走り、数十秒から数分のオーバーヘッドが発生します。
*   **補足:** `files/` ディレクトリのコピーアップは、このカーネル実体化（ステップ1）よりも後（ステップ4）に行われるため、「ユーザーがUpperで追加した独自設定を `initramfs` に反映させる」という高機能化（カスタマイズ性の向上）のメリットは得られていません。現状の再生成は、純粋に「正しいデプロイフックを走らせて抽象化を保つため」のコストとして割り切られています。
